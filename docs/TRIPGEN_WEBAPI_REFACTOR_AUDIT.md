## Flow Summary

`TripGenerator_WebAPI_refactor` is the mainline WebAPI-based trip+trajectory generator. For each person's activity chain, it:
1. Computes OD distance
2. Routes via local DRM (Dijkstra) AND optionally queries CSIS WebAPI for mixed/public-transit route
3. Picks the cheapest mode via cost-minimization (`determineTransportMode`)
4. Generates trip records (`person.addTrip`) and trajectory points (`person.addTrajectory`)

This combines what the legacy pipeline does in two separate steps (TripGenerator + TrajectoryGenerator) into a single pass.

## A. Mode-Choice Consistency (`determineTransportMode`, lines 229–271)

### A1. Cost model for WALK uses road route length, not walk distance — **CORRECT but aggressive**
- Walk time = `route.getLength() / 1.38` (1.38 m/s ≈ 5 km/h). Uses Dijkstra car-route length as proxy for walk distance.
- For long distances this overestimates walk attractiveness because the road route may be shorter than the pedestrian path, but in practice the high FATIGUE_INDEX_WALK (1.5) compensates. **Acceptable approximation.**

### A2. BICYCLE time = walktime / 2 — **inconsistent with legacy Speed class**
- Legacy `Speed.java`: walk = 4.8 km/h, bike = 15 km/h (ratio 3.125:1).
- Here: bike = 2x walk speed. Bicycles are **undervalued** (made to appear slower than they are), biasing mode choice away from cycling.
- **Severity: MEDIUM** — calibration issue, not a bug.

### A3. CAR cost includes `FARE_INIT=200` — walk/bike do not
- This correctly penalizes car for short trips. But `FARE_INIT` is applied per-trip, not per-day. A person making 4 car trips pays 800 yen in init costs. Whether this matches behavioral intent is unclear.
- **Severity: LOW** — by design, but worth documenting intent.

### A4. MIX time unit inconsistency — **BUG**
- Line 258: `mixedtime = mixedResultsHolder[0].get("total_time").asDouble()` — comment says "Travel time from WebAPI is in minute"
- Line 259: `mixedcost = mixedfare + mixedtime / 60 * FARE_PER_HOUR`
- If `total_time` is in **minutes**, then `mixedtime / 60` converts to **hours**. This is correct.
- But compare with line 237: `roadtime = route.getCost()` — this is in **seconds** from Dijkstra, then `roadtime / 3600 * FARE_PER_HOUR` converts to hours. Also correct.
- **No unit bug here** — both convert to hours before multiplying by FARE_PER_HOUR. The comment is accurate.

### A5. Empty choices map → `ETransport.NOT_DEFINED` — **silent fallthrough**
- If `route == null` (Dijkstra fails) AND `distance <= MIN_TRANSIT_DISTANCE` (no MIX query): choices is empty, `orElse(NOT_DEFINED)`.
- In `process()` (line 538), NOT_DEFINED falls to the `else` branch (lines 538–549) which creates a stay-at-home stub with 5-minute duration. The person is effectively stuck.
- **No crash**, but **no warning logged** that Dijkstra failed for a >0 distance trip. A `System.err.println` would help.
- **Severity: MEDIUM** — silent wrong result.

### A6. `CAR_AVAILABILITY = 0.4` — non-deterministic mode choice
- `getRandom() < 0.4` means 40% of non-car-owners can still choose car. This is evaluated per-trip, not per-person. A person might drive to work but not drive home. The legacy `TripGenerator` uses roulette probabilities (also per-trip), so this is consistent in spirit but means **inconsistent round-trip mode** is possible.
- **Severity: LOW** — by design, but differs from `refactor2` which uses 0.25.

## B. Mixed-Transport Consistency (`handleMixedTransport`, lines 299–313)

### B1. `mixedResultsHolder[0]` index — **CORRECT for refactor, differs from refactor2**
- `refactor` stores MIX result at index `[0]` (the only element in the 1-element array).
- `refactor2` uses a 2-element array: `[0]` = road route, `[1]` = mixed route.
- This is internally consistent within each file. No cross-contamination bug.

### B2. `endTime += mixedTime` has no downstream consumer — **dead computation**
- Line 301: `endTime += mixedTime` modifies a local copy (primitive `long` parameter).
- The actual end time used for trajectory stamping comes from `addTimeStampedSubpoints(nodes, startTime, endTime, ...)` on line 311, which DOES use the updated value. So this is correct — the updated `endTime` is consumed.

### B3. `handleMixedTransport` does NOT add the trip records — trips come from `processMixedTransportFeatures`
- Line 310: `processMixedTransportFeatures(...)` calls `addTripForSubtrip(...)` for each segment.
- Line 311: `addTimeStampedSubpoints(...)` generates trajectory points.
- **Trips and trajectory are generated by different data paths**: trips from WebAPI feature segments, trajectory from `extractNodesFromMixedResults` (which uses railway Dijkstra for station-to-station segments).
- This means trip sub-leg OD coordinates come from WebAPI feature coordinates, while trajectory nodes come from local DRM/railway network routing. **These may not match exactly** (different road/rail networks), but this is acceptable as a design tradeoff.

### B4. The final station segment is never processed — **BUG**
- `extractNodesFromMixedResults` (lines 332–395): The commented-out block at lines 386–389 would process the last station segment. Without it, if the route **ends** with a train segment, the last station-to-station rail Dijkstra is never run.
- Lines 386–389: `// if (firstStationNode != null) { processStationSegment(...) }`
- This means the final rail segment's detailed nodes are omitted from the trajectory. The trajectory will jump from the last non-station node to... nothing (or the next activity location via the caller's logic).
- **Severity: HIGH** — trajectory is incomplete for routes ending with rail.

### B5. `nodes.size()==0` check (lines 390–393) — debug stub, not a guard
- When nodes is empty, prints "error is here" and "stop!" but still returns the empty list.
- Upstream, `addTimeStampedSubpoints` calls `TrajectoryUtils.putTimeStamp(nodes, ...)` with empty list — behavior depends on TrajectoryUtils implementation but likely produces an empty map/no points.
- **Severity: MEDIUM** — should be a proper warning or error, not a debug print.

## C. Trip Record Integrity (`addTripForSubtrip`, lines 427–457)

### C1. Mode detection logic (lines 433–444) — **overly complex, potential misclassification**
```java
int firstMode = currentSubtrip.get(0).path("properties").path("transportation").asInt();
boolean allSame = true, hasMode2 = false, hasMode3 = false;
for (JsonNode node : currentSubtrip) {
    int transportation = node.path("properties").path("transportation").asInt();
    if (transportation != firstMode) allSame = false;
    if (transportation == 2) hasMode2 = true;
    if (transportation == 3) hasMode3 = true;
}
int finalMode = allSame ? firstMode : (hasMode2 ? 2 : (hasMode3 ? 3 : lastMode));
```
- `currentSubtrip` starts with `prevNode` from the **previous** segment (line 419 in `processMixedTransportFeatures`). So `currentSubtrip.get(0)` is the last node of the preceding segment, which may have a different `transportation` value than the current segment.
- This means `firstMode` may not reflect the actual first mode of the current segment, and `allSame` will often be false due to this boundary node.
- When `allSame` is false: mode 2 (TRAIN) takes priority over mode 3 (BUS) over `lastMode`. This heuristic is reasonable but could misclassify a short walk segment between two train segments as TRAIN if any node in the subtrip happens to have mode 2.
- **Severity: MEDIUM** — the boundary-node contamination means the first node's mode is from the wrong segment.

### C2. `mode == ETransport.CAR && publicTransit` → WALK (line 446–448)
- When WebAPI returns `transportation=1` (which maps to CAR via the default case in `getTransport`), and the overall route has transit stations, force it to WALK.
- This is a reasonable heuristic: WebAPI mode 1 likely means "walking to/from station" (access/egress), not driving. The constant mapping `default → CAR` in `getTransport` is the root issue — mode 1 from WebAPI probably means "walk" or "unspecified foot travel", not car.
- **Severity: MEDIUM** — the workaround is correct but masks a mapping error. WebAPI mode codes should be documented.

### C3. `depTime` is the same for all sub-trips — **BUG**
- In `processMixedTransportFeatures` (lines 401–425), `currentTime` is computed but only used locally — `addTripForSubtrip` receives `currentTime` as `depTime`, but the commented-out line 452 (`depTime += distance/speed`) in `addTripForSubtrip` means **all sub-trips get the same departure time**.
- In `refactor2` (line 365), `depTime += (long)(distance / 8.33)` is active, incrementing departure time per sub-leg.
- **Severity: HIGH** — all sub-trips within a mixed route share the same `depTime`, making temporal ordering incorrect.

### C4. `currentTime` computation in `processMixedTransportFeatures` (lines 410–415)
- When `lastMode == 1`: `currentTime += 300` (5 minutes flat)
- Otherwise: `currentTime += mixedResultsHolder[0].path("total_transport_time").asLong() * 60`
- The `total_transport_time` is added for **every** non-mode-1 segment, not just the current segment's transport time. This accumulates the **total** transit time once per segment boundary, which is wrong — it should be the segment's own transport time.
- **Severity: HIGH** — `total_transport_time` is the entire route's transit time, not per-segment. If there are 3 transit segments, the time is tripled.

## D. Trajectory Integrity

### D1. Double `addTrajectory` call — **BUG**
- Line 529: `points.addAll(subpoints)` — adds subpoints to the main points list.
- Line 564: `person.addTrajectory(points)` — adds all accumulated points at the end of the activity loop.
- But ALSO: line 549: `person.addTrajectory(points)` in the NOT_DEFINED fallback branch.
- If a person has 3 activities and the 2nd trip resolves but the 3rd falls to NOT_DEFINED, `addTrajectory(points)` is called at line 549 (partial) AND again at line 564 (full). Since `addTrajectory` does `this.trajectory.addAll(points)`, the same points are added **twice**.
- **Severity: HIGH** — trajectory duplication. Every NOT_DEFINED trip causes all prior trajectory points to be doubled.

### D2. NOT_DEFINED branch creates 2-point trajectory at the **previous** location
- Lines 543–548: Creates SPoints at `pre.getLocation()` (the previous activity's location, not the current one). This puts the person at the wrong location for a 5-minute window.
- Since `pre = next` is set at line 561 AFTER this block, and the trajectory from line 564 would include these mislocated points, the person appears to teleport.
- **Severity: MEDIUM** — wrong location in trajectory, but only for edge cases where Dijkstra fails.

### D3. Trajectory mode is always `ETransport.MIX` for mixed routes
- `addTimeStampedSubpoints` passes `nextMode` (= `ETransport.MIX`) for all nodes, even walk/bus/train sub-legs. The trajectory doesn't reflect per-segment mode.
- Compare: the trip records DO get per-segment modes via `addTripForSubtrip`.
- **Severity: LOW** — trajectory mode granularity is coarser than trip mode. Downstream consumers (MeshVolumeCalculator) don't use trajectory mode for filtering, so impact is minimal.

### D4. `configureCalendar` adds 45 years and 9 months — **intentional epoch shift**
- Lines 220–227: Shifts dates by +45y9m. This converts from the internal time representation (seconds since midnight as a small epoch) to a real calendar date. Both refactor and refactor2 do this identically.
- **No bug** — this is a known convention. But it means trajectory timestamps are in a future year (circa 2070 if base is 2025), which may confuse downstream tools expecting real dates.

### D5. `startTime * 1000` conversion (line 460/524)
- `new Date(startTime * 1000)` assumes startTime is in seconds. `startTime = next.getStartTime()` which is loaded from activity CSV. Activity times are in seconds-since-midnight (0–86400 range). This is correct.

## E. Cost/Fare/Time Model Consistency

### E1. Parameter comparison across variants

| Parameter | refactor | refactor2 | Legacy TripGenerator |
|-----------|----------|-----------|---------------------|
| FARE_PER_KILOMETER | 10 | 51 | N/A (uses Speed) |
| FARE_PER_HOUR | 1000 | 1000 | N/A |
| FATIGUE_INDEX_WALK | 1.5 | 2.5 | N/A |
| FATIGUE_INDEX_BICYCLE | 1.2 | 2.0 | N/A |
| FARE_INIT | 200 | 75 | N/A |
| CAR_AVAILABILITY | 0.4 | 0.25 | N/A |
| MIN_TRANSIT_DISTANCE | 1000 | 1000 | N/A |
| Walk speed | 1.38 m/s | via road API×600×10 | 4.8 km/h (1.33 m/s) |
| Bike speed | walk/2 | road API×60×3 | 15 km/h |
| Car speed | Dijkstra cost | road API | 20 km/h |

- `refactor` walk speed (1.38 m/s ≈ 4.97 km/h) is close to legacy (4.8 km/h). OK.
- `refactor` bike speed (2.76 m/s ≈ 9.9 km/h) is much slower than legacy (15 km/h). This makes bikes less competitive.
- `refactor2` walk time = `roadtime * 600 * 10` = `roadtime * 6000` — if roadtime is in **minutes** from API, this produces seconds. But `roadtime / 3600` then converts to hours. This gives `roadtime_min * 6000 / 3600 = roadtime_min * 1.67` hours of walk. This seems extremely wrong (a 30-min drive → 50 hours of walking). **refactor2 has a severe walk-cost bug** that refactor avoids.

### E2. Road route cost vs WebAPI cost
- `refactor` uses **local Dijkstra** for car/walk/bike costs and **WebAPI** for mixed-mode costs. This is a good design — it avoids an API call per trip for non-transit modes.
- `refactor2` calls **WebAPI road route** for every trip, even car/walk. This is expensive and rate-limited.
- **refactor is the better design** for production use.

### E3. `AppDate` hardcoded to `"20240401"` (line 590)
- All WebAPI mixed-route queries use April 1, 2024 as the reference date. This affects transit schedules/fares.
- Should be configurable or derived from simulation date.
- **Severity: LOW** — affects fare accuracy but not structural correctness.

## F. Edge Cases / Silent Wrong-Result Risks

### F1. `createSession()` returns empty string on failure (line 159)
- If session creation fails (non-200), `sessionId = ""`. All subsequent API calls use `Cookie: WebApiSessionID=` with no value. They'll likely fail with 401/403, and `getMixedRoute` will return an empty JsonNode.
- The `getMixedRoute` failure path (lines 694–701): `mapper.readTree("")` returns a `MissingNode`. Then `path("num_station").asInt()` returns 0, so `publicTransit = false`, and MIX is never offered.
- **Net effect**: session failure silently disables all public transit. No error is raised.
- **Severity: HIGH** — complete silent degradation of the primary feature.

### F2. `connManager.setDefaultMaxPerRoute(100)` > `setMaxTotal(32)` — **misconfigured**
- Max per route (100) exceeds max total (32). The per-route limit is capped by the total, making it effectively 32. Not a bug but misleading config.
- **Severity: INFO**

### F3. `httpClient` shared across threads — **thread-safety concern**
- `PoolingHttpClientConnectionManager` is thread-safe by design. `CloseableHttpClient` with pooling manager is also thread-safe. **No bug.**
- However, the `httpClient` is never closed (no `try-with-resources` or `finally` block), so connections may leak on abnormal termination.
- **Severity: LOW**

### F4. `route.listNodes()` / `route.listLinks()` size mismatch assumption
- Line 294: `assignLinksToSubpoints(subpoints, links)` assumes `links.size() == subpoints.size() - 1` (N nodes → N-1 links). This is the standard Dijkstra route structure.
- But if `subpoints` was populated from a different source (e.g., a partial route), an `IndexOutOfBoundsException` could occur at `subpoints.get(k)`.
- **Severity: LOW** — only fails if Dijkstra returns inconsistent route data.

### F5. Per-city file `new TripGenerator_WebAPI_refactor(...)` creates a new HTTP session per city file
- Line 823: Constructor called inside the file loop. Each city file creates a new WebAPI session.
- For 43 files in a prefecture, this creates 43 sessions. If session creation is rate-limited, this could cause failures.
- **Severity: MEDIUM** — should create one session per prefecture run, not per city file.

### F6. `file.getName().substring(9, 14)` — fragile filename parsing
- Lines 812–814: Assumes activity filenames follow `person_XXXXX.csv` (9 chars prefix, 5 char city code). If filenames differ (e.g., different prefix length from the new `ActivityGenerator`), this will extract wrong characters or throw StringIndexOutOfBoundsException.
- Activity file naming convention: `person_XXXXX.csv` where XXXXX is the 5-digit city code. The unified `ActivityGenerator` writes with the original filename, so this should be OK.
- **Severity: LOW** — brittle but currently correct.

### F7. `process()` returns 0 always, even on exception
- Line 565: `catch (Exception e)` catches all exceptions, prints stack trace, but still returns 0. The `error` counter in `call()` checks `res < 0`, which never happens.
- **Severity: MEDIUM** — exceptions are swallowed per-person, making error tracking unreliable.

### G3. Key design improvements in `refactor` over `refactor2`
1. Local Dijkstra for car/walk/bike avoids unnecessary API calls
2. Railway network Dijkstra for station-to-station trajectory interpolation
3. Fail-fast exception handling in `call()` and `generate()`
4. Env-var credential loading (not hardcoded)

### G4. Regressions in `refactor` vs `refactor2`
1. Sub-trip departure time is NOT incremented (C3)
2. `total_transport_time` accumulation bug (C4)
3. Last rail segment not processed (B4)

## Flow Invariants

1. **Every person gets at least one trip record** — guaranteed by the `activities.size() == 1` branch (line 480)
2. **Every person gets trajectory points** — guaranteed by `addTrajectory(points)` at line 564, BUT can be doubled (D1)

5. **Trajectory points are temporally ordered** — guaranteed by `TrajectoryUtils.putTimeStamp` for single-mode; may break at MIX/non-MIX boundaries

## Suggested High-Value Tests

1. **Trajectory duplication test**: Create a person with 3 activities where the 3rd trip has distance=0 (triggers NOT_DEFINED). Verify trajectory point count equals expected (not doubled).

2. **Mixed-mode sub-trip timing test**: Create a MIX route with 3 segments (walk→train→walk). Verify each sub-trip's `depTime` is strictly increasing and sums to total travel time.

3. **Session failure graceful degradation test**: Mock `createSession()` to return `""`. Verify that mode choice falls back to car/walk/bike without exceptions, and that a warning is logged.

4. **Final rail segment test**: Provide a WebAPI response where the last feature has a station name. Verify that the rail Dijkstra is run for the final station-to-station segment (currently fails — B4).

5. **Round-trip mode consistency test**: Generate trips for a worker with HOME→OFFICE→HOME pattern. Verify the return trip mode is reasonable (not guaranteed same-mode by design, but should not be wildly different, e.g., car outbound, walk 30km return).
