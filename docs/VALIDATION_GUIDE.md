# Validation Guide

## Running validation

```bash
# Full prefecture validation (activity + trip + trajectory)
scripts/validate/run_validation.sh <pref_code> <output_root> [report_dir]

# Examples
scripts/validate/run_validation.sh 22 /tmp/pflow_smoke
scripts/validate/run_validation.sh 22 /tmp/pflow_webapi_smoke /tmp/reports/22
```

Default report directory: `<output_root>/validation/<pref>/`.

Individual validators can also be run standalone:

```bash
python3 scripts/validate/validate_activity.py <activity_dir> [output.json]
python3 scripts/validate/validate_trip.py <trip_dir> [output.json] [activity_dir]
python3 scripts/validate/validate_trajectory.py <trajectory_dir> [output.json]
python3 scripts/validate/validate_summary.py <report_dir> <pref_code> [output.md]
```

The trip validator accepts an optional `activity_dir` argument for person-ID cross-referencing.

## Expected directory structure

```
<output_root>/
  activity/<pref>/        activity_XXXXX.csv  (one per city)
  trip/<pref>/            trip_XXXXX.csv      (one per city)
  trajectory/<pref>/      trajectory_XXXXX.csv
```

Missing directories are skipped with a SKIP message. At least one directory must exist.

## Output files

Per-run reports are written to `<report_dir>/`:

| File | Contents |
|------|----------|
| `activity.json` | Per-file validation results |
| `trip.json` | Per-file validation results |
| `trajectory.json` | Per-file validation results |
| `summary.md` | Aggregate Markdown report |

Each JSON file contains a `files` array with per-file `status`, `errors`, `warnings`, and `stats`.

## What each validator checks

### Activity (`validate_activity.py`)

CSV columns (no header): person_id, age, gender_id, labor_id, start_time, duration, purpose_id, lon, lat, gcode.

| Check | WARN | FAIL |
|-------|------|------|
| Rows with <10 columns | Any | — |
| Row count | — | < 10 |
| Coordinates outside Japan bbox (120-155E, 20-50N) | <= 5% | > 5% |
| Unknown purpose_id | Any | — |
| Unknown gender_id | Any | — |
| Empty gcode | > 1% | — |
| start_time out of range (0..172800) | Any | — |
| Non-monotonic start_time per person | > 10% of persons | — |
| Max activities per person > 30 | Yes | — |

Stats reported: row_count, num_persons, avg/max activities per person, purpose distribution, gender distribution.

### Trip (`validate_trip.py`)

CSV columns (no header): person_id, dep_time, origin_lon, origin_lat, dest_lon, dest_lat, transport_id, purpose_id, labor_id.

| Check | WARN | FAIL |
|-------|------|------|
| Rows with <9 columns | Any | — |
| Row count | — | < 10 |
| Coordinates outside Japan bbox | <= 5% | > 5% |
| Placeholder NOT_DEFINED (zero-distance) | Any | — |
| Unexpected NOT_DEFINED (nonzero-distance) | <= 5% | > 5% |
| Single mode > 95% of real-movement trips | Yes | — |
| Trips exceeding 2000 km | Any | — |
| dep_time out of range (0..172800) | Any | — |
| Non-monotonic dep_time per person (> 20%) | Yes | — |

Stats reported: row_count, num_persons, mode_share (count + pct per mode), not_defined breakdown, real_movement_mode_share, distance median/p95, person coverage vs activity (when cross-referenced).

### Trajectory (`validate_trajectory.py`)

CSV columns (no header): person_id, unix_time_ms, datetime_str, lon, lat, transport_id, purpose_id, labor_str, link_id.

| Check | WARN | FAIL |
|-------|------|------|
| Rows with <8 columns | Any | — |
| Row count | — | < 10 |
| Coordinates outside Japan bbox | <= 5% | > 5% |
| Timestamp monotonicity violations | — | Any |
| Spatial jumps > 50 km | <= 1% | > 1% |
| Speed > 500 km/h between consecutive points | Any | — |
| Missing link_id | > 30% | — |
| Duplicate rows (same pid + timestamp + coords) | > 1% | — |

Stats reported: row_count, num_persons, avg points per person, transport distribution, missing_link_ratio, persons_with_issues.

## PASS / WARN / FAIL rules

Each file gets a single status:

- **PASS** — no errors and no warnings
- **WARN** — at least one warning, no errors
- **FAIL** — at least one error

The prefecture summary reports aggregate counts across all files: total PASS, WARN, FAIL. Overall status is FAIL if any file FAILs.

## NOT_DEFINED transport classification

Transport mode 0 (`NOT_DEFINED`) is split into two categories:

- **Placeholder** (zero-distance): origin and destination are within 1 meter. These represent stay-at-home stubs or return-to-same-location markers generated by the activity model. They are expected and always WARN-only.

- **Unexpected** (nonzero-distance): origin and destination differ, indicating real movement was assigned no transport mode. These signal a bug or calibration gap. Thresholds: WARN > 1%, FAIL > 5%.

Console output shows both counts per file: `ND:755p+0u` means 755 placeholder, 0 unexpected.

## Mode share metrics: primary vs secondary

The trip validation summary reports **two different mode-share metrics** at different aggregation levels. They are not interchangeable.

### [PRIMARY] Trip-level mode share (rep_mode)

This is the metric that is **directly comparable to `data/tuning/transport_share_targets.csv`** and is the one to use when judging:
- Whether tuned parameters are working
- Whether mode split realism is acceptable
- Production quality vs. PT survey ground truth

**How it is computed**: deduplicate trip rows on `(person_id, trip_id)`, then count each unique trip once using its `rep_mode` value (column 11, 12-column format). A multi-leg transit trip like `walk → bus → walk` counts as **1 BUS trip**.

### [SECONDARY] Subtrip-level mode distribution (transport_id)

This is the raw per-row count of the `transport_id` column (column 6). Useful for:
- Counting individual bus/train segments
- Analyzing transfer structure
- Segment-level routing diagnostics

**Not directly comparable to tuning targets**: a multi-leg transit trip like `walk → bus → walk` counts as **2 WALK + 1 BUS = 3 rows**, inflating WALK share.

### Why the two differ

For pref 22 with config_003, the same data produces:

| Mode | Trip-level (primary) | Subtrip-level (secondary) |
|------|---------------------|---------------------------|
| CAR | 57.3% | 56.9% |
| WALK | 23.4% | 23.5% |
| BICYCLE | 14.4% | 14.3% |
| NOT_DEFINED | 4.7% | 4.7% |
| BUS | 0.2% (6 trips) | 0.6% (20 segments) |
| TRAIN | 0.0% (1 trip) | 0.1% (2 segments) |

The BUS/TRAIN gap shows each transit trip averages ~3 subtrip segments (access walk + transit + egress walk).

**Rule**: When comparing to tuning targets, use the PRIMARY table. When analyzing transit coverage, use the SECONDARY table.

### NOT_DEFINED breakdown (still subtrip-level)

- **Placeholder** (zero-distance): stay-at-home / return-to-same-location markers. Always WARN-only.
- **Unexpected** (nonzero-distance): real movement with no mode assigned. WARN > 1%, FAIL > 5%.

Console output per file: `ND:755p+0u` = 755 placeholder, 0 unexpected.

## Prefecture 22 baseline (2026-03-11)

### Legacy pipeline (full prefecture, all cities)

| Stage | Files | PASS | WARN | FAIL |
|-------|-------|------|------|------|
| Activity | 43 | 43 | 0 | 0 |
| Trip | 129 | 0 | 129 | 0 |
| Trajectory | 1668 | 159 | 56 | 1453 |

**Activity**: 3.47M persons, 13.6M rows. All PASS. Purpose mix: HOME 54.6%, FREE 15.0%, OFFICE 11.4%.

**Trip**: 2.88M persons, 6.9M rows. All WARN due to placeholder NOT_DEFINED (847K, 12.3% — all zero-distance). Zero unexpected NOT_DEFINED. Real-movement mode share: CAR 60.7%, WALK 24.3%, BICYCLE 13.3%, TRAIN 1.7%.

**Trajectory**: 1453 of 1668 files FAIL due to timestamp monotonicity violations. All FAIL files are from the **legacy TrajectoryGenerator** (file pattern `trajectory_XXX_Y_0001_NNNNNN.csv` — batched output). Root cause: the time-exceeded retry loop recalculates routes without re-sorting the timeline. This is deferred legacy technical debt, not a mainline blocker.

### WebAPI mainline (city 22429, 2000 persons)

| Stage | Files | PASS | WARN | FAIL |
|-------|-------|------|------|------|
| Activity | 2 | 2 | 0 | 0 |
| Trip | 2 | 0 | 2 | 0 |
| Trajectory | 2 | 0 | 0 | 2 |

**Trip**: 756 NOT_DEFINED all placeholder (12.5%), 0 unexpected. Real-movement mode share: CAR 52.5%, WALK 29.0%, BICYCLE 18.5%.

**Trajectory**: 191 monotonicity violations in 226K points (0.08%). Breakdown: 129 NOT_DEFINED return-home stubs (cosmetic), 11 trip boundary overlaps, 51 same-mode within-trip. No spatial jumps or speed violations. Tracked as calibration item, not a hard blocker.

### Interpretation

- Trip WARNs from placeholder NOT_DEFINED are expected and benign — these are structural artifacts of the activity model (stay-at-home persons get a zero-distance NOT_DEFINED trip).
- The non-commuter `_n` files show ~35% placeholder NOT_DEFINED, which is consistent with the higher share of single-activity (home-only) persons in the non-commuter population.
- Trajectory FAILs in legacy output are a known limitation. The mainline WebAPI pipeline produces far fewer violations (0.08% vs widespread in legacy).
